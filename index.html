M5 Cross-Operator Coordination and Communication Protocol Architecture

Multi-Operator Fleet Coordination for Live Institutional Validator Infrastructure
Phase: M5: Live Validator Operations Classification: Protocol Architecture Specification Version: 1.0, March 2026 Alignment: Superstructure Validator Platform
Document Purpose This specification defines the coordination and communication protocols that allow multiple operators to function as a coherent validator fleet without any operator gaining visibility into another's custody state, signing infrastructure, or institutional relationships. Every coordination mechanism described here routes through the Middleware Layer (Layer 3). No direct operator-to-operator communication channel exists at any point in these protocols. This architecture extends the zero-custody boundary established in M2 and the operator isolation model established in M3 into the multi-operator coordination domain, which is the final architectural prerequisite before the validator network can scale beyond its founding cohort.

Inter-Operator Communication Boundaries and Permitted Channels

The fundamental rule governing inter-operator communication is that no operator communicates directly with another operator at any layer of the stack. All information that one operator needs about the state of another flows through the OCP coordination layer, which acts as the sole intermediary. This is not a policy preference: it is an architectural requirement. A direct channel between operators would create a surface through which one operator could probe, influence, or gain visibility into another's operational or custody state. The prohibition is enforced at the network layer, not just by policy.

1.1 Permitted Coordination Channels

Three coordination channels exist. All three are OCP-mediated. None connect operators to each other.

ChannelChannel IDDirectionOperator AccessData PermittedTransport
OCP Admin Channel CHAN_OCP_ADMIN Operator to OCP; OCP to Operator (bidirectional per operator, isolated) Each operator has an independent, isolated admin channel instance. Operator A cannot read Operator B's admin channel. Maintenance declarations, patch acknowledgments, capacity reports, remediation plans, incident responses mTLS 1.3, operator client certificate; certificate scoped to operator identity
OCP Coordination Broadcast CHAN_OCP_BROADCAST OCP to all operators (one-way broadcast; operators cannot write to this channel) Read-only for all operators simultaneously; each operator receives the same payload; no operator-specific filtering Upgrade notifications, governance proposals, network status events, fleet-wide alerts, rollback signals mTLS 1.3 with OCP server certificate; operators authenticate as subscribers
OCP Telemetry Channel CHAN_OCP_TELEMETRY Operator to OCP (one-way upload; OCP does not return operator telemetry to other operators) Write-only for operators; OCP aggregates telemetry from all operators but does not expose individual operator telemetry to peers Health metrics, heartbeat signals, ceremony completion reports, binary integrity check results mTLS 1.3; payload signed with operator telemetry signing key (scoped, per M5 Telemetry Framework)
No Fourth Channel Permitted Any communication path between operators that does not route through one of these three OCP-mediated channels is prohibited. This includes: peer-to-peer messaging between operator nodes, shared configuration repositories accessible by multiple operators, out-of-band coordination via external services (Slack, email, shared dashboards), and any form of operator-to-operator API. Detection of an unauthorized inter-operator communication path is treated as a P1 security incident per M5 Security Hardening escalation procedures.

1.2 Permitted and Prohibited Data Types

Even within the permitted channels, data types are explicitly enumerated. Anything not on the permitted list is not permitted. Ambiguous data types default to prohibited.

Permitted Data Types (OCP-Mediated Only)

  • Validator node identifier (public registry entry)
  • Operator identifier (public registry entry)
  • Health tier classification (healthy, degraded, critical, offline)
  • Upgrade version number and binary manifest hash (broadcast only)
  • Maintenance window declaration (timing and duration)
  • Governance proposal payload (OCP-constructed, broadcast)
  • Quorum vote result (aggregated by OCP; individual votes not shared)
  • Network participation rate (aggregate; not per-operator breakdown)
  • Ceremony completion count (aggregate; not per-operator breakdown)
  • Failover handoff notification (validator ID, handoff window, no custody data)
  • Rollback signal (affects all operators; not targeted at individuals)

Prohibited Data Types (Even via OCP)

  • Any key material, HSM reference, or custody system identifier
  • Per-operator signing ceremony logs or ceremony success rate
  • Per-operator telemetry data or health score
  • Operator infrastructure topology (IP addresses, host counts, region)
  • Individual operator CRS score or routing weight allocation
  • Operator identity of the institution relationship counterpart
  • Signing envelope content (payload or signed blob)
  • Operator incident history or security alert details
  • Binary version currently running on a specific operator's nodes
  • Per-operator capacity or resource utilization metrics
  • Any data that could allow one operator to fingerprint another's infrastructure
Aggregate vs. Individual Data Distinction The distinction between aggregate and individual data is structural, not statistical. Network participation rate (aggregate: 97.3% of all validators fulfilled slots this cycle) is permitted because it contains no information about any specific operator. Per-operator participation rate (Operator B fulfilled 94.1% of slots) is prohibited because it reveals information about Operator B's infrastructure to Operator A. OCP enforces this distinction at the data aggregation layer before broadcast: individual operator data never enters the broadcast payload.

1.3 Isolation Guarantees

Isolation is enforced at four levels to ensure that the communication boundary holds even under adversarial conditions.

Isolation LevelMechanismWhat It Prevents
Network isolation Operator nodes have no direct network reachability to other operator nodes. Firewall allowlists permit only OCP endpoint addresses. Any connection attempt to a non-OCP address generates a P3 alert. Direct TCP/IP connections between operator infrastructure.
Channel isolation Each operator's admin channel instance is authenticated with an operator-specific mTLS certificate. OCP validates certificate identity on every request and rejects cross-operator certificate use. Operator A using Operator B's credentials to read Operator B's admin channel.
Data aggregation isolation OCP aggregates telemetry and health data before any broadcast. The aggregation layer strips operator identifiers from individual data points before computing network-wide figures. Per-operator data leaking into broadcast payloads that all operators receive.
Custody isolation Signing ceremony infrastructure (Layer 1: institution) is never referenced in any inter-operator coordination message. Handoff notifications carry validator identity only, not custody system references. One operator inferring which HSM or custody system another operator's institution uses.

Middleware-Mediated Coordination Architecture

The middleware coordination layer is the OCP component that brokers all cross-operator communication. It is the only entity in the architecture that holds a complete view of the operator fleet: which operators are active, what health states each node is in, what governance proposals are pending. This concentrated visibility is what makes middleware-mediation powerful and what makes it a security-critical component. The design constrains what the middleware does with that visibility just as carefully as it constrains what operators can do.

2.1 Intermediary Service Design

The OCP coordination layer runs three distinct sub-services, each with a narrowly scoped function. No sub-service has access to data outside its defined scope.

Sub-ServiceService IDScopeData AccessOutput
Fleet State Aggregator OCP_FSA Collects telemetry from all operators, computes aggregate fleet health metrics, maintains per-node health tier state Read: all operator telemetry channels. Write: OCP metric store. No write access to broadcast channel. Aggregate fleet health records; per-node health tier transitions; inputs to OCP health scoring engine (M5 Telemetry Framework)
Coordination Broadcast Service OCP_CBS Constructs and distributes broadcast payloads to all operators simultaneously; enforces data type restrictions before broadcast Read: OCP_FSA aggregate outputs only (not individual telemetry). Write: broadcast channel. No access to per-operator admin channels. Upgrade notifications, governance proposals, fleet alerts, rollback signals; all payloads signed with OCP broadcast signing key

Write: per-operator admin channels (targeted notifications). No access to telemetry or custody data. The three sub-services do not share a database or a runtime process. Each reads from a defined set of OCP data stores with read-only access scoped to its function. A compromise of OCP_CBS does not grant access to per-operator admin channel data because OCP_CBS has no read access to admin channels. Separation at the data access layer means a partial middleware compromise does not automatically compromise all coordination functions.

2.2 Direct Operator-to-Operator Restriction Model

The restriction model is enforced at four layers. All four must be active simultaneously. Enforcement at any single layer is not sufficient, because a sufficiently determined operator could attempt to bypass a single-layer restriction.

Layer 1: Network Firewall Enforcement

Operator nodes are on isolated network segments with allowlist-only egress per M5 Security Hardening Section 1.2. The allowlist contains OCP endpoints only. No operator node IP address range appears in any other operator's allowlist. Network-layer enforcement means even a misconfigured operator process cannot accidentally reach another operator's infrastructure.

Layer 2: OCP Certificate Authority Control

All OCP-mediated connections use mTLS with certificates issued by the OCP certificate authority. OCP CA issues operator certificates scoped to specific channel types: an admin channel certificate cannot authenticate to the telemetry endpoint, and vice versa. The OCP CA does not issue certificates that would authorize operator-to-operator connections, because no such connection type is defined in the certificate policy. A certificate presented for an undefined connection type is rejected without logging the attempt as a valid authentication event.

Layer 3: OCP Application-Layer Authorization

Every request to any OCP endpoint is validated against an authorization policy that specifies permitted operations per operator identity. The policy is managed by OCP, not by operators. An operator cannot modify their own authorization scope. Requests for operations outside the defined scope return ERR_OP_NOT_AUTHORIZED and are logged to the OCP audit trail with the requesting operator identity and the attempted operation.

Layer 4: Data Aggregation Firewall

The OCP_CBS broadcast service enforces the data type restrictions from Section 1.2 before any payload is constructed. Individual operator identifiers are stripped from all data before aggregation output is generated. A broadcast payload is checked against the prohibited data type list before transmission. Any payload containing a prohibited data type is rejected and the operator who triggered the payload generation receives ERR_DATA_PROHIBITED through their admin channel.

Fleet-Level Upgrade Coordination

Fleet-level upgrades present a coordination challenge specific to the zero-custody architecture: the signing binary that validators run must be updated across all operators without requiring operators to coordinate directly, without synchronizing signing ceremony windows across institutions, and without leaving the network in a state where some validators run an incompatible binary version and cannot participate in signing with others.

3.1 Rolling Upgrade Propagation

Upgrades propagate in three waves, with OCP holding the gate between each wave. Operators do not know which wave they are in until OCP notifies them; the wave assignment is made by OCP based on operator health score, patch currency, and fleet diversity to ensure the network retains sufficient healthy validators throughout the upgrade.

Wave 0: Upgrade Authorization (Institution Signing Ceremony)

OCP constructs a binary upgrade authorization payload: new binary manifest hash, version identifier, compatibility matrix, and rollback binary hash. This is a root-gated action. OCP delivers signing ceremony envelopes to all participating institutions simultaneously via the standard M3 ceremony pathway. Each institution applies Maker/Checker/Reviewer process independently. Upgrade authorization requires quorum institution sign-off (threshold defined in Section 5.1) before any binary is distributed to operators. No operator receives the new binary before institution quorum is confirmed. Wave 0 has no time pressure: it runs on institution ceremony timelines, not operator maintenance windows.

Wave 1: First 25% of Operators (4-Hour Observation Window)

OCP selects the Wave 1 operator set using three criteria: operators with the highest current health score (least risk of compounding an existing problem), operators with the most diverse validator types relative to each other (to limit correlated failure), and operators who have most recently completed successful patch cycles (freshest patch currency signal). OCP notifies Wave 1 operators via admin channel with a 2-hour preparation window before the upgrade is pushed. Operators confirm readiness through the admin channel. OCP pushes the new binary manifest to Wave 1 nodes. Secure boot verification confirms binary hash against the manifest on each node before the process starts. OCP monitors Wave 1 for 4 hours before opening Wave 2.

Wave 2: Next 50% of Operators (2-Hour Observation Window)

Wave 2 proceeds only if Wave 1 passes the observation gate: all Wave 1 nodes must be in healthy or degraded tier after 4 hours, no P1 or P2 incidents attributed to the upgrade, and ceremony success rate on Wave 1 nodes is at or above pre-upgrade baseline. If the gate passes, OCP notifies Wave 2 operators. The observation window for Wave 2 is 2 hours (compressed relative to Wave 1 because Wave 1 data provides confidence). Emergency upgrades use a 1-hour Wave 1 observation window as specified in M5 Security Hardening fleet patch procedures.

Wave 3: Remaining Operators (Completion Confirmation)

After Wave 2 gate passes, remaining operators receive the upgrade. OCP generates a fleet attestation report after all operators complete: confirms every active node is running the authorized binary version per their registered manifest hash. Report is delivered to all institutions via the institutional gateway and retained in the OCP immutable audit store. Operators can verify their own binary version via admin channel attestation query; they cannot query other operators' versions.

3.3 Rollback Coordination

Rollback is triggered by OCP when Wave 1 or Wave 2 observation gates fail. No individual operator can trigger a fleet-wide rollback; only OCP can issue a fleet rollback signal through the broadcast channel. An operator can roll back their own node independently via the admin channel (self-rollback), but self-rollback does not affect other operators.

Synchronized Signing Window Avoidance The rolling wave design explicitly prevents the need for synchronized signing windows across operators during upgrades. Because each wave upgrades a subset of the fleet while others remain on the previous version, ceremonies continue to route to non-upgrading operators throughout the upgrade window. The version compatibility check ensures that mixed-version fleets can still process signing ceremonies: OCP routes ceremony requests to operators running a compatible version during the transition period. No institution is asked to hold or reschedule a signing ceremony because of an upgrade in progress on another operator's node.

Cross-Operator Failover Handoff Protocols

A failover handoff transfers validator operational responsibilities from one operator to another. Two handoff types exist: planned maintenance handoffs (initiated by the outgoing operator) and unplanned outage handoffs (initiated by OCP when a node enters offline tier). Both types preserve the zero-custody architecture throughout. The institution's signing keys stay in the institution's custody environment regardless of which operator is currently running the validator. The handoff transfers operational responsibility, not custody.

4.1 Planned Maintenance Handoff

Step 1: Maintenance Declaration (Outgoing Operator, T-48h minimum)

The outgoing operator submits a maintenance declaration via CHAN_OCP_ADMIN specifying: validator node identifier(s) affected, planned maintenance window start and end times, and the nature of the maintenance (routine, upgrade, hardware replacement, decommission). The declaration does not name the incoming operator; OCP selects the receiving operator from the active operator pool based on current health score and available capacity. The outgoing operator has no visibility into which operator OCP selects. OCP acknowledges the declaration and registers a maintenanceDeclared flag in the OCP node registry for the affected validators.

Step 2: Receiving Operator Selection and Notification (OCP, T-24h)

OCP selects the receiving operator using the same criteria as Wave 1 upgrade selection: highest health score among eligible operators with sufficient capacity headroom. The receiving operator is notified via their admin channel with: the validator node identifier, the handoff window timing, and configuration parameters needed to stand up the replacement node (software version, network configuration, endpoint addresses). No custody information is included. The receiving operator does not learn the identity of the outgoing operator from this notification; they learn only the validator identity and operational parameters.

Step 3: Parallel Readiness Verification (OCP, T-2h)

The receiving operator stands up a warm standby node and confirms readiness to OCP via admin channel. OCP verifies the standby node through secure boot attestation (binary hash matches authorized manifest), telemetry continuity (at least 5 consecutive heartbeat cycles from the standby node), and health score (standby node must be in healthy tier). The institution is notified via gateway that a handoff is scheduled; no institution action is required for planned maintenance handoffs.

Step 4: Weight Transfer (OCP, at handoff window start)

OCP executes the routing weight transfer: outgoing operator's weight is set to 0 for the affected validator; receiving operator's weight increases by the transferred amount, subject to the 60% single-operator cap. The transfer is atomic within the OCP routing engine: there is no window where the validator has zero weight across both operators. Signing ceremonies for the affected validator are suspended for a maximum of 30 seconds during the weight transfer. If the 30-second suspension would create a ceremony backlog that threatens network throughput, OCP routes pending ceremonies to other validators in the fleet and clears the backlog before executing the transfer.

Step 5: Recovery Confirmation (OCP, T+30m post-handoff)

OCP monitors the receiving operator's node for 30 minutes after the weight transfer completes. Confirmation criteria: node remains in healthy tier for 30 minutes, ceremony success rate on the new node meets or exceeds the pre-handoff baseline, and no P1 or P2 incidents are generated. After confirmation, OCP closes the maintenance record and notifies both operators via their respective admin channels (outgoing operator receives a completion confirmation; receiving operator receives a full-activation confirmation). Institution receives a handoff completion notice via gateway.

4.2 Unplanned Outage Handoff

Unplanned outage handoffs are initiated automatically by OCP when a validator node enters offline tier per the M5 Telemetry Health Scoring Framework. The protocol is compressed relative to planned handoffs because there is no preparation window.

4.3 Custody Boundary Preservation During Handoff

The zero-custody boundary does not change during a handoff. The following invariants hold throughout every handoff type.

Custody Invariants: Maintained at All Times Including During Handoff (1) The institution's private signing keys remain in the institution's custody environment (HSM, air-gapped system, or institutional multisig) regardless of which operator is running the validator. Keys are never transferred between operators. Keys are never transferred to OCP. (2) The incoming operator receives no information about the outgoing operator's infrastructure, signing ceremony history, or custody system configuration. (3) Signing ceremony envelopes are constructed by OCP and delivered to the institution; they are not relayed through the outgoing or incoming operator. (4) If the handoff is due to a security incident, forensic evidence from the outgoing operator's node is preserved per the M5 Security Hardening chain-of-custody procedure before any process on that node is terminated.

Governance Signal Propagation Architecture

Governance decisions affecting the operator fleet (parameter changes, operator additions and removals, policy updates, protocol upgrades) must propagate to all operators in a way that is verifiable, auditable, and resistant to unilateral authority by any single participant. The governance propagation architecture extends the coordination model from M3 Founding Cohort Governance into multi-operator fleet operation, where the cohort may grow beyond the founding members.

5.1 Quorum Requirements by Decision Type

50%+1
Simple Majority

66%

Unanimous
Operator removal; changes to custody boundary definitions; modifications to the zero-custody architecture

Quorum is calculated across active institutions, not across operators. An institution with multiple active validators counts as one vote. This prevents an operator running many validators from having proportionally more governance influence than an operator running fewer. Quorum thresholds apply to voting institutions, not to operator count: a 66% threshold requires 66% of enrolled institutions to sign, regardless of how many validator nodes those institutions collectively authorize.

Quorum Calculation During Reduced Cohort If one or more institutions are in an offline or critical validator state at the time a governance vote is called, the quorum denominator is the count of institutions with at least one healthy or degraded validator, not the total enrolled institution count. This prevents a governance deadlock where a single offline institution blocks a required unanimous vote indefinitely. The quorum adjustment is logged in the OCP governance record for the vote. Any institution that was excluded from a quorum calculation due to validator state receives a notification and may submit a dissenting record to the governance log within 72 hours of the vote completing.

5.2 Governance Signal Propagation Protocol

Stage 1: Proposal Origination (OCP Governance Service)

Governance proposals originate from two sources: OCP (for protocol-level changes like upgrades or parameter adjustments) or institutions (for operator-level changes like additions or removals). Operator-originated proposals are not permitted: operators submit requests to OCP through the admin channel, and OCP evaluates whether to formalize the request as a governance proposal. An operator requesting their own removal or another operator's removal submits a formal request to OCP; OCP constructs the proposal independently if the request meets threshold criteria. This prevents an operator from constructing a self-serving governance payload and broadcasting it directly to institutions.

Stage 2: Proposal Broadcast (OCP_CBS, Simultaneous Delivery)

OCP constructs the governance proposal payload: proposal identifier (UUID), proposal type and quorum threshold required, proposal content, voting deadline, and a proposal content hash (SHA-256). The payload is signed with OCP's governance signing key. OCP_CBS delivers the payload to all operators via the broadcast channel simultaneously. Operators cannot respond to the proposal: their role in governance propagation is to ensure their validator nodes remain operational so their institution can vote. The institution receives the proposal directly through the institutional gateway, not relayed through the operator.

Stage 3: Institution Independent Evaluation and Signing

Each institution evaluates the proposal through its internal Maker/Checker/Reviewer process. The Reviewer role is non-waivable for all governance votes. Institutions vote by submitting a signed vote envelope via the standard signing ceremony pathway: OCP constructs the vote envelope, the institution signs inside their custody environment, and OCP receives the signed blob. Institutions cannot see how other institutions have voted before the voting deadline. OCP withholds vote tally information until the deadline passes. This prevents a bandwagon effect where late voters adjust their position based on early results.

Stage 4: Quorum Determination and Result Broadcast

After the voting deadline, OCP tallies signed votes, verifies each vote envelope's signature and freshness, and determines whether the quorum threshold was met. The result payload contains: proposal identifier, final vote count (yes/no/abstain), quorum threshold applied, quorum met (yes/no), and the list of institutions that voted (not how they voted: vote direction is private; participation is public). The result is broadcast via CHAN_OCP_BROADCAST to all operators and delivered to all institutions via the gateway. If quorum was met, OCP schedules the implementation action.

Stage 5: Implementation (Root-Gated Action for Applicable Decision Types)

For decisions that require a root-gated action (operator deregistration, protocol upgrade, custody boundary change), OCP constructs the implementation signing ceremony envelope after quorum confirmation. The envelope references the governance proposal identifier so institutions know exactly which vote outcome they are authorizing at the implementation stage. Institutions that voted against a passed proposal still receive the implementation ceremony; they may decline to sign, which generates a governance dispute record. The dispute resolution process is defined in Section 5.4.

Unilateral Authority Prevention No single party holds unilateral authority over any governance outcome. OCP can originate proposals but cannot vote on them; OCP does not hold a signing key that counts toward institution quorum. Institutions can vote but cannot originate proposals unilaterally; a single institution cannot call a vote that OCP refuses to formalize. Operators have no governance standing: they cannot originate proposals, vote, or dispute outcomes. The cohort's governance authority is distributed across institutions with the quorum thresholds in Section 5.1 as the mathematical guarantee that no minority can impose a decision on the majority, and no majority can permanently exclude a minority without clearing the relevant quorum bar.
